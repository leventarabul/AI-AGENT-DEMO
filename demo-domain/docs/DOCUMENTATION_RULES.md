# Documentation Rules

## Language Policy

All documentation in this project must be written in **English**, regardless of the language used by contributors or developers.

### Guidelines

- All markdown files (.md) must be in English
- All code comments must be in English
- All file names and directory names must be in English
- All configuration files must use English descriptions and comments
- All error messages and logging should be in English

### User Communication

While users and developers may communicate in Turkish or other languages, all documentation that becomes part of the codebase must be translated to or written in English.

---

## Medium Publication Guidelines

Documentation created for this project may be published on Medium to share knowledge and implementation details with the broader community.

### Publication Standards

- All content must follow the Language Policy (English only)
- Include clear, step-by-step instructions for reproducibility
- Add architecture diagrams or visual explanations where helpful
- Include code snippets with proper syntax highlighting
- Reference GitHub repository links to the full codebase
- Add a "Getting Started" section for new readers
- Include prerequisites and system requirements
- Provide troubleshooting section for common issues
- Add links to related articles or resources

### Article Structure

1. **Introduction** - Problem statement and solution overview
2. **Architecture** - System design and components
3. **Setup Instructions** - Step-by-step implementation guide
4. **Code Walkthrough** - Key code sections explained
5. **Testing** - How to verify the implementation works
6. **Conclusion** - Summary and next steps

### Content Types

- Technical deep-dives on system components
- Setup and configuration guides
- Integration tutorials
- Lessons learned and best practices
